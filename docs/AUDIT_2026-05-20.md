# Codebase audit — 2026-05-20

Cross-app pass on the deployed state at `github.com/mahakaal02/bet`. Driven by four parallel Explore agents covering backend, bet, auctions+admin, and aviator+helm+android.

This document is **the running list** of findings. Items shipped as fixes in this PR (or PR #27) are marked ✅; everything else is a backlog item for the team to triage.

---

## CRITICAL (fix before any public launch)

### 1. ✅ Schema drift on `Auction.startsAt`
- **Where**: [`backend/prisma/schema.prisma`](../backend/prisma/schema.prisma) declared the column and `@@index([status, startsAt])`, but **no prior migration ever created either**. PR #26's seed migration was the first thing to read `startsAt` and hit `column does not exist`, leaving the migration row stuck in `_prisma_migrations` and blocking every subsequent deploy.
- **Runtime impact (also fixed)**: [`AuctionsService.promoteUpcomingToLive()`](../backend/src/auctions/auctions.service.ts) filters on `startsAt: { not: null, lte: now }` — would have thrown the same `column does not exist` error the moment any UPCOMING auction was created.
- **Fix**: see PR #27 — prepends `ALTER TABLE … ADD COLUMN IF NOT EXISTS` and `CREATE INDEX IF NOT EXISTS` to the seed migration so it self-patches.

### 2. ✅ `prisma-migrate` init container stays stuck on a single failed migration
- **Where**: [`helm/kalki/templates/backend.yaml`](../helm/kalki/templates/backend.yaml) + `bet.yaml`. A failed migration writes a row to `_prisma_migrations` with `finished_at IS NULL`. Subsequent `migrate deploy` invocations refuse to proceed — every future deploy stays blocked until manual `prisma migrate resolve` from inside the pod.
- **Fix**: PR #27 adds an idempotent pre-cleanup step (`DELETE FROM _prisma_migrations WHERE finished_at IS NULL AND … started_at < NOW() - INTERVAL '5 minutes'`) so a corrected migration retries on the next pod boot. 5-minute threshold avoids racing genuinely in-flight migrations.

### 3. Postgres password is a literal default checked into git
- **Where**: [`helm/kalki/values.yaml`](../helm/kalki/values.yaml) `postgres.password: "kalki-postgres-default"`.
- **Risk**: anyone with repo read access has prod DB credentials. OWASP A02.
- **Why this PR doesn't auto-fix**: the running StatefulSet's PV is *already* initialised with that exact password — the `POSTGRES_PASSWORD` env var is ignored on subsequent boots. Rotation must happen via in-cluster `ALTER USER postgres WITH PASSWORD …` first, then the in-cluster `kalki-helm-values` Secret needs the new value, then the literal can be replaced with a `CHANGE_ME_*` sentinel + a chart-fail guard.
- **Action**: ✅ rotation runbook added in `values.yaml` comments; the actual rotation is operator-only work. A follow-up PR can add the fail-fast `{{- fail … }}` guard once the live cluster is on a non-default password.

### 4. Admin SPA stores JWT in unencrypted `localStorage`
- **Where**: [`admin/src/lib/auth.ts:13-18`](../admin/src/lib/auth.ts).
- **Risk**: any XSS payload reads admin tokens; admin role is verified from a `localStorage` user blob (`isAdmin`) the attacker can edit. Two CRITICAL audit findings combine here.
- **Fix path** (backlog): switch admin auth to httpOnly cookies (backend sets cookie on login), and add a server-side `/admin/whoami` check on app init that confirms `isAdmin` from the backend. Larger refactor — not in this PR.

### 5. Android `usesCleartextTraffic="true"` globally enabled
- **Where**: [`app/src/main/AndroidManifest.xml:16`](../app/src/main/AndroidManifest.xml).
- **Risk**: MITM on prod if a reverse-proxy ever serves HTTP. Token in cleartext.
- **Fix path** (backlog): replace with a `network_security_config.xml` that allows cleartext *only* for `localhost` and `10.0.2.2` (emulator), HTTPS-only elsewhere. Conditional `android:networkSecurityConfig` per build type.

### 6. Android JWT in plaintext `SharedPreferences`
- **Where**: [`app/src/main/java/com/uniquebid/app/data/auth/TokenStore.kt:18,23-24`](../app/src/main/java/com/uniquebid/app/data/auth/TokenStore.kt). Comment in source already flags this as a TODO.
- **Fix path** (backlog): swap for `EncryptedSharedPreferences` (`androidx.security:security-crypto`) — Android Keystore-backed encryption at rest.

## HIGH

### 7. Razorpay webhook endpoint not implemented on backend
- **Where**: backend `src/payments/` references a webhook in comments but no controller exists. The only verification path is the client-side `POST /payments/verify` flow.
- **Risk**: if the client's network drops between payment + verify, the credit is lost. Server-side webhook would catch it.
- **Fix path** (backlog): add `POST /payments/webhook` with Razorpay's signed webhook secret verification (separate from payment-signature HMAC).

### 8. Bid placement & wallet debit cross service boundary, no compensation log
- **Where**: [`backend/src/bids/bids.service.ts:98-125`](../backend/src/bids/bids.service.ts) — insert bid → debit Bet wallet via HTTP. On debit failure, the local bid is deleted. But if the HTTP response is lost mid-flight (network partition after Bet committed), the local rollback fires and the user loses coins.
- **Fix path** (backlog): outbox pattern — write the wallet-debit intent into a local table in the same Prisma transaction as the bid insert, then a background worker drains the outbox with retries. Reconciliation job for orphan bids.

### 9. ✅ Aviator double-tap can fire two `placeBet` / `cashout` requests
- **Where**: [`aviator/components/BetControls.tsx`](../aviator/components/BetControls.tsx). React state `busy` updates async; two rapid taps in the same frame both see `busy=false` in their closure.
- **Fix**: synchronous `inFlightRef` (`useRef`) gates re-entry before any async work. Server-side bet-already-placed check is still the authority, but the client now never sends the duplicate.

### 10. ✅ Aviator wallet balance can drift on bet error + on socket reconnect
- **Where**: [`aviator/lib/useAviator.ts`](../aviator/lib/useAviator.ts) only re-fetched balance on cashout success and on GAME_CRASH. A bet that errors out, or a brief socket disconnect, could leave the displayed balance behind the server's.
- **Fix**: re-fetch authoritative balance on every socket `connect` event + on any `placeBet` error.

### 11. ✅ Backend login + register endpoints had no per-IP throttle
- **Where**: [`backend/src/auth/auth.controller.ts`](../backend/src/auth/auth.controller.ts). Only the global `@nestjs/throttler` cap covered them.
- **Fix**: `@Throttle({ login: { limit: 8, ttl: 60_000 } })` + `register: 3/min`. Pairs with the existing bcrypt(10) cost (~80 ms/check) to keep brute-force economically infeasible.

### 12. NO_WINNER ringmaster cascade can be exhausted
- **Where**: `backend/src/auctions/auctions.service.ts:178-199`. Cascade loop caps at 200 iterations. A user could in principle stuff bids fast enough to exhaust the cap before the next ringmaster collision lands, briefly producing a LOWEST_UNIQUE winner against intent.
- **Fix path** (backlog): increment the ringmaster's collision amount by `+0.01` on each cascade so it can't itself be collided, or use a Redis lock to serialise ringmaster placements.

### 13. `db push --accept-data-loss` fallback in prod init container
- **Where**: same Helm templates as #2. The `|| db push --accept-data-loss` branch can DROP columns silently if `schema.prisma` ever loses one.
- **Why this PR doesn't remove**: the noisy alternative (pod fails to start on any migration error) means an outage every time a migration is broken. With PR #27's failed-migration cleanup, the fallback is now mostly dead code, but removing it should pair with operator alerting on migration failure. Backlog.

## MEDIUM

### 14. Auction list `include: { winner: … }` is a Prisma join, not N+1
- **Where**: [`backend/src/auctions/auctions.service.ts:29-52`](../backend/src/auctions/auctions.service.ts). The audit flagged this as N+1; Prisma 5 actually batches `include` for one-to-many into a single SQL query with a LEFT JOIN. Verified — this is fine as-is.

### 15. `listBids()` is unbounded but legitimately needs full history
- **Where**: same file, line 318. The classification logic requires the entire bid history to compute `statusAtPost` per row. Pagination would break correctness. Leave as-is unless an admin auction grows to >10k bids and needs streaming.

### 16. Aviator: socket auto-reconnect re-syncs via server snapshot
- **Where**: aviator gateway pushes a `STATE_SNAPSHOT` on every `connect` automatically. The client's `STATE_SNAPSHOT` handler applies it. So mid-round reconnect IS already covered — the audit finding here was a false alarm. Verified.

### 17. ✅ Auctions login page used to show seed credentials as placeholders
- **Where**: [`auctions/app/login/LoginForm.tsx`](../auctions/app/login/LoginForm.tsx). Shipped in PR #23 and PR #25 — placeholders dropped, labels read literal `mobile/email` / `password`.

### 18. WhatsApp OTP send is a no-op
- **Where**: `backend/src/auth-whatsapp/auth-whatsapp.service.ts` — `sendOtp()` logs the code in dev but doesn't call any provider in prod. A user can sign up but never receives the OTP. Backlog: wire Meta WhatsApp Cloud API or Twilio.

### 19. Admin destructive actions only confirm via `window.confirm()`
- **Where**: `admin/src/pages/Auctions.tsx` — close/start/delete. Trivially spoofable, no CSRF token. Backlog: replace with a "type DELETE to confirm" modal; add CSRF middleware to backend admin routes.

### 20. Admin client-side file size validation only
- **Where**: `admin/src/pages/ImageManager.tsx:67` — `accept=` filter is the only check; server-side mime sniffing missing. Backlog.

### 21. No image-vulnerability scanning in build workflow
- **Where**: `.github/workflows/build-and-push.yml` — pushes to Docker Hub without a scan. Backlog: Trivy/Scout step that fails on high/critical CVEs.

### 22. `npm test … --passWithNoTests` lets the backend ship without tests
- **Where**: `.github/workflows/backend-and-admin-ci.yml:44`. Remove the flag once at least one Jest test exists on master. Backlog.

### 23. `next lint` is `continue-on-error: true` on bet-ci
- **Where**: `.github/workflows/bet-ci.yml:78-90`. ESLint isn't wired up at all; the step is decoration. Wire it up, or remove the step. Backlog.

- bcrypt rounds are env-driven on email auth (`BCRYPT_ROUNDS`, default 10) but hardcoded `10` in WhatsApp auth.
- Achievement list endpoint on Bet has no `take` cap (low-impact — catalog is ~10 entries).
- Admin SPA hardcoded `VITE_API_BASE_URL` → `/api` fallback works in dev only.
- Multiple admin forms missing `htmlFor` labels (a11y).
- Auctions middleware token-via-query-param: redirect immediately strips the token, no real referer leak risk in practice. Audit flagged it; verified safe.
